# vm2.DevOps — Comprehensive Review (2026-03-03)

## Executive Summary

vm2.DevOps is a well-architected CI/CD automation toolkit for .NET NuGet packages. The three-layer
design (consumer workflows → reusable workflows → bash scripts) with a shared library foundation is
clean and scalable. The documentation is thorough — unusually so for a DevOps project.

Below: strengths, bugs, warnings, architecture suggestions, documentation improvements, and
COPILOT_INSTRUCTIONS.md guidance.

---

## 1. Architecture & Design

### Strengths

- **Clean layering** — consumer → reusable → script separation keeps concerns well-isolated.
- **Consistent naming convention** — `UPPER_SNAKE` → `lower-kebab` → `lower_snake` transformation
  is well-documented and consistently applied.
- **Three-file script pattern** — `script.sh` / `script.usage.sh` / `script.args.sh` provides
  excellent uniformity and discoverability.
- **Dual-layer NuGet caching** with weekly rotation is pragmatic.
- **Prerelease guard** in the release version calculation prevents version regression.

### Architectural Concerns

1. **Single branch reference (`@main`)** — All consumer workflows pin to `@main`. A breaking change
   in vm2.DevOps immediately affects every consumer. Consider:
   - Tagging stable releases of vm2.DevOps itself (e.g., `@v2`)
   - Or documenting the risk and the rollback procedure explicitly

2. **Sparse checkout fragility** — The two-directory sparse checkout
   (`scripts/bash/lib` + `.github/actions/scripts`) is an implicit contract. If a script adds a new
   dependency directory, all consumers silently break. Document this as a maintenance risk with a
   checklist for adding new script directories.

3. **`RELEASE_PAT` as admin bypass** — The branch protection ruleset relies on the PAT owner being
   a bypass actor. If the PAT leaks, an attacker can push directly to `main`. Highlight this risk
   more prominently.

## 2. Bugs Found

### 🔴 BUG-1: `_benchmarks.yaml` — Cache `restore-keys` prefix never matches

**File:** `vm2.DevOps/.github/workflows/_benchmarks.yaml` (~line 92)

```yaml
restore-keys: |
  bencher-cli-${{ runner.os }}-latest-
```

Primary key is `bencher-cli-Linux-20260302`. The restore prefix `bencher-cli-Linux-latest-` will
**never** match. Fix:

```yaml
restore-keys: |
  bencher-cli-${{ runner.os }}-
```

### 🔴 BUG-2: `run-benchmarks.sh` — `warning` piped to `to_summary` loses variable in subshell

**File:** `vm2.DevOps/.github/scripts/run-benchmarks.sh`

**Status:** Verify this is fully fixed. In earlier rounds the pattern was:

```bash
warning_var tfm "..." "net10.0" | to_summary
```

The pipe runs `warning_var` in a subshell, so `tfm` assignment is lost. The fix is two separate
lines:

```bash
warning_var tfm "..." "net10.0"
echo "⚠️ Using default TFM: $tfm" | to_summary
```

**Action:** Grep all `.sh` files for any remaining `variable_assignment | pipe` patterns:

```bash
grep -rn '.*_var.*|' vm2.DevOps/.github/scripts/
grep -rn '.*_var.*|' vm2.DevOps/scripts/
```

### 🔴 BUG-3: `CI.yaml` template — `get-params` output name mismatch

**File:** `.github/workflow-templates/CI.yaml`

The `get-params` job outputs section may use `reset-thresholds` while the step writes
`reset_benchmark_thresholds` (which GitHub converts to `reset-benchmark-thresholds`). Then
`call-ci` reads `needs.get-params.outputs.reset-thresholds` which would be empty.

**Action:** Verify all output names align:

```text
step writes:      reset_benchmark_thresholds
GitHub converts:  reset-benchmark-thresholds
job outputs:      reset-benchmark-thresholds    ← must match
downstream reads: reset-benchmark-thresholds    ← must match
```

### 🔴 BUG-4: `validate-input.args.sh` — `--reset-benchmark-thresholds` flag vs value

**File:** `vm2.DevOps/.github/scripts/validate-input.args.sh`

If `--reset-benchmark-thresholds` is parsed as a flag (no `shift`) but is called with a value
argument:

```bash
--reset-benchmark-thresholds '${{ inputs.reset-benchmark-thresholds }}'
```

Then `'true'` or `'false'` becomes the next `$1` and hits the `*) unknown option` case.

**Fix:** Either:

- Make it an option with a value (`shift` after reading), OR
- Keep it as a flag and remove the value from the caller, using conditional inclusion:

  ```yaml
  ${{ inputs.reset-benchmark-thresholds == true && '--reset-benchmark-thresholds' || '' }}
  ```

**Recommendation:** Make it an option with a value — booleans-as-flags are messy in CI.

### 🔴 BUG-5: Potential subshell variable loss in other scripts

**Action:** Search all scripts for the pattern `cmd_that_sets_variable | another_cmd`:

```bash
# Find all pipes where the left side might be setting variables
grep -rn 'warning_var\|error_var\|info_var' vm2.DevOps/ | grep '|'
```

Also check any `while read ... | ...` patterns:

```bash
grep -rn '| *while' vm2.DevOps/
grep -rn '| *read' vm2.DevOps/
```

## 3. Warnings

### 🟡 WARN-1: `_benchmarks.yaml` — `--branch-start-point-hash` empty on push events

```yaml
--branch-start-point-hash "${{ github.event.pull_request.base.sha || '' }}"
```

On `push` events, `github.event.pull_request.base.sha` is empty. Make it conditional:

```bash
PR_BASE_SHA="${{ github.event.pull_request.base.sha }}"
if [[ -n "$PR_BASE_SHA" ]]; then
    BENCHER_ARGS+=(--branch-start-point-hash "$PR_BASE_SHA")
fi
```

### 🟡 WARN-2: `_ci.yaml` — `fromJSON` on potentially empty string

```yaml
reset-benchmark-thresholds: ${{ fromJSON(needs.validate-input.outputs.reset-benchmark-thresholds) }}
```

If `reset-benchmark-thresholds` is ever empty or malformed, `fromJSON` will fail the workflow.
This is consistent with how other parameters are handled, but worth noting.

### 🟡 WARN-3: `run-benchmarks.sh` — TFM search order

Ensure the search order is:

1. Project file `<TargetFramework>` (singular)
2. Project file `<TargetFrameworks>` (plural)
3. `Directory.Build.props` `<TargetFramework>` (singular)
4. `Directory.Build.props` `<TargetFrameworks>` (plural)

Project file must always take precedence over `Directory.Build.props`.

### 🟡 WARN-4: CHANGELOG.md (vm2.Ulid) — formatting issues

- **Duplicate `[1.0.6]` section** (lines ~52 and ~56)
- **`v1.0.9` and `v1.0.8`** use `v` prefix but lower entries use `[1.0.x]` without `v` — inconsistent
- **`v1.0.9` and `v1.0.8`** say "See prereleases below" but no prerelease sections are visible

### Suggested DevOps-specific sections to add

#### Bash Scripting Conventions

- All scripts must pass ShellCheck with no warnings.
- Use `set -euo pipefail` at the top of every script entry point.
- Quote all variable expansions: `"$var"` not `$var`.
- Use `[[` for conditionals, not `[`.
- Prefer `local` for function-scoped variables.
- Use `readonly` or `declare -r` for constants after final assignment.
- Follow the three-file pattern: `script.sh`, `script.usage.sh`, `script.args.sh`.
- Source `gh_core.sh` for CI scripts; source `core.sh` for utility scripts.
- Use library functions (`info`, `error`, `warning`, `trace`) instead of raw `echo`.
- Use `confirm` for interactive prompts; CI scripts should use `--quiet` mode.
- **Never pipe a command that sets variables** — the left side of `|` runs in a subshell
  and variable assignments are lost. Use separate lines or process substitution.
- Accumulate validation errors with `|| true` and call `exit_if_has_errors` at the end to
  report all errors at once.

#### GitHub Actions Workflow Conventions

- Pin third-party actions to specific SHA hashes or major versions (e.g., `actions/checkout@v4`).
- All reusable workflows use `workflow_call` trigger.
- Use `workflow_dispatch` inputs only for human-facing overrides.
- Document all inputs, secrets, and permissions in the workflow file comments.
- Use concurrency groups to prevent parallel runs: `ci-${{ github.workflow_ref }}`.
- Never expose secrets in logs — use `::add-mask::` when necessary.
- Prefer `$GITHUB_OUTPUT` over deprecated `::set-output::`.
- Boolean inputs from `workflow_dispatch` are strings — always compare with `== 'true'`.

#### Adding a New CI Script

1. Create three files: `script.sh`, `script.usage.sh`, `script.args.sh`.
2. Source `gh_core.sh` at the top of `script.sh`.
3. Define `usage()` in `script.usage.sh` following existing patterns.
4. Define argument parsing in `script.args.sh` using the library pattern.
5. Add the script to SCRIPTS_REFERENCE.md documentation.
6. Add a calling workflow in `_*.yaml` if needed.
7. If the script needs files from vm2.DevOps, update the sparse checkout directories.

#### Variable Naming Convention

- GitHub Actions inputs/outputs: `lower-kebab-case`
- Bash variables: `lower_snake_case`
- Environment variables: `UPPER_SNAKE_CASE`
- The transformation `UPPER_SNAKE` → `lower-kebab` → `lower_snake` is automatic via
  `args_to_github_output` and the argument parsing framework.
